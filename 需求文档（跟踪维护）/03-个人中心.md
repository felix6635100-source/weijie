## 0. 模块概览

「个人中心」是承载账号体系、登录注册、个人资料维护、隐私设置的全局模块。入口位于：

- App 启动后未登录态 → 任意 Tab 的"登录引导卡"或顶部头像 → P1 登录。
- 已登录态 → 一级页面顶部头像 → P5 个人设置。
- 已登录态 → 一级页面顶部 🔔 消息中心按钮（v2.1 起 · v2.5 起为唯一入口）→ P7 消息中心。

## P1. 登录主页

全屏页面（覆盖 Tab Bar），承载登录注册全流程。所有子步骤（P1.1 / P1.2 / P2 / P3 / P4）在同一容器内通过表单切换，共用顶栏。

### 状态

P1 本身只是容器；具体内容由当前激活的子步骤决定（详见各子步骤章节）。

### 元素构成

1. **公共顶栏**：
   - 左：✕ 关闭（默认）/ ‹ 返回（仅在 P2 验证码子步骤显示）。
   - 右：「帮助」文字按钮。
2. **子步骤容器**：根据当前 step 渲染 P1.1 / P1.2 / P2 / P3 / P4 之一。

### 操作逻辑

- 点击 ✕ → 关闭整个登录页，回到调起前的 Tab。
- 点击 ‹ → 仅在 P2 显示，回到上一个子步骤。
- 点击"帮助" → 跳帮助 H5 页。

## P1.1. 短信验证码登录（默认子步骤）

App 默认进入的登录入口。

### 状态

| 状态     | 触发条件                       | 视觉                          |
| ------ | -------------------------- | --------------------------- |
| 默认     | 进入 P1                      | 主按钮置灰禁用，提示协议默认勾选            |
| 手机号有效  | 输入 11 位且符合格式 + 协议勾选        | 主按钮高亮可点                     |
| 协议未勾选  | 用户取消勾选                     | 主按钮置灰；点击主按钮触发抖动 + 轻提示       |
| 加载中    | 点击主按钮后等待短信下发              | 主按钮内 spinner，文字保留           |

### 元素构成

1. **标题**："欢迎登录 AIRing"（28/700）。
2. **手机号输入行**：
   - 左：国家区号"+86" + ▾。
   - 中：分隔符。
   - 输入框：11 位数字限制，占位"请输入手机号"。
   - 右：✕ 一键清空（仅有内容时显示）。
3. **提示文案**：灰字"未注册的手机号验证后自动创建 AIRing 账号"。
4. **协议同意行**：
   - 左：方形勾选框（默认勾选）。
   - 文案："我已阅读并同意《用户协议》和《隐私政策》"。
5. **主按钮**（通栏品牌蓝）："获取短信验证码"。
6. **次链接**（居中文字）："密码登录"。

### 操作逻辑

- 点击区号 → 弹国家区号选择器（本期写死 +86，仅展示动效）。
- 输入手机号 → 实时校验格式，主按钮联动启用 / 禁用。
- 点击 ✕ → 清空手机号输入。
- 点击协议勾选行 → 切换勾选态。
- 点击《用户协议》/《隐私政策》 → 跳对应 H5 页。
- 点击主按钮（高亮可点态） → 调用短信下发接口 → 切换到 P2 验证码输入子步骤。
- 点击主按钮（协议未勾选） → 协议条抖动 1 次 + 弹轻提示"请先阅读并同意协议"。
- 点击"密码登录" → 切换到 P1.2 子步骤。

### 业务规则

- 手机号格式：1 开头 11 位数字（13/14/15/16/17/18/19 段）。
- 60 秒内同一手机号最多请求 1 次验证码。

## P1.2. 密码登录（子步骤）

### 状态

| 状态     | 触发条件                  | 视觉                |
| ------ | --------------------- | ----------------- |
| 默认     | 从 P1.1 切换进入            | 主按钮置灰禁用            |
| 可登录    | 手机号 + 密码非空 + 协议勾选     | 主按钮高亮可点           |
| 错误     | 提交后服务端返回错误            | 显示红字"手机号或密码错误，请检查后重试" |
| 加载中    | 点击登录后等待响应             | 主按钮内 spinner       |

### 元素构成

1. **标题**："密码登录"。
2. **手机号输入行**（同 P1.1）。
3. **密码输入行**：
   - 输入框：占位"请输入登录密码"，默认隐藏。
   - 右侧 👁 切换显示 / 隐藏。
4. **错误提示行**（红字，仅出错时显示）："手机号或密码错误，请检查后重试"。
5. **协议同意行**（同 P1.1）。
6. **主按钮**（通栏）："登录"。
7. **底部三联次链接**（横排）：
   - 左："验证码登录"。
   - 右："忘记密码？"。

### 操作逻辑

- 输入手机号 / 密码 → 主按钮联动。
- 点击 👁 → 切换密码可见。
- 点击主按钮 → 提交登录；成功 → 弹 P-LOGIN-DONE 轻提示后跳上次所在 Tab；失败 → 显示错误行 + 抖动一次。
- 点击"验证码登录" → 切回 P1.1。
- 点击"忘记密码？" → 进入 P-FORGOT 流程（自动跳到 P1.1 但带"忘记密码"上下文）。

### 业务规则

- 同手机号连续登录失败 5 次 → 锁定 10 分钟，弹轻提示"尝试次数过多，请稍后再试"。

## P2. 验证码输入（子步骤）

### 状态

| 状态  | 触发条件         | 视觉                       |
| --- | ------------ | ------------------------ |
| 默认  | P1.1 提交后进入   | 6 格空格 + 倒计时 60 秒          |
| 输入中 | 输入 1~5 位数字   | 已输入格位填值 + 当前格位高亮          |
| 错误  | 输入 6 位但服务端不通过 | 红字"验证码错误，请重新输入" + 自动清空    |
| 倒计时结束 | 60 秒后        | 倒计时变成"重新获取"按钮，可点         |
| 已注册成功 | 6 位正确 + 服务端识别为已注册 | 自动跳走，弹 P-LOGIN-DONE        |
| 未注册成功 | 6 位正确 + 服务端识别为未注册 | 自动切到 P3 设置密码（注册流程）        |

### 元素构成

1. **标题**："输入验证码"。
2. **副标题**：动态显示"验证码已发送至 +86 138 0000 0000"。
3. **6 位验证码格子行**：
   - 6 个等宽方格，每格显示一位数字。
   - 当前焦点格底边高亮。
   - 隐藏的输入框接收键盘输入并自动分配到 6 格。
4. **错误行**（红字，仅出错时）："验证码错误，请重新输入"。
5. **元信息行**（横排）：
   - 左：动态文字"重新获取 (60s)" / 倒计时结束后变成可点按钮"重新获取"。
   - 右：链接"手机号已停用？"。

### 操作逻辑

- 输入数字 → 自动填入下一格。
- 6 位输完 → 自动提交校验。
- 校验错误 → 显示红字 + 自动清空 + 焦点回到第 1 格。
- 倒计时归零后点击"重新获取" → 重新下发短信，倒计时重置为 60 秒。
- 点击"手机号已停用？" → 跳人工申诉表单（TBD-7）。

### 业务规则

- 验证码 6 位数字，5 分钟内有效。
- 同一手机号 24 小时内最多请求 5 次。
- iOS / Android 自动填充短信验证码。

## P3. 设置登录密码（子步骤）

### 使用场景

- **注册**：短信验证码通过后，手机号在系统中无账号。
- **忘记密码**：忘记密码流程的最后一步。

### 状态

| 状态     | 触发条件         | 视觉                       |
| ------ | ------------ | ------------------------ |
| 默认     | 进入此子步骤       | 主按钮置灰；两条规则提示均为灰圆圈        |
| 长度满足   | 密码 1 满足 6-20 位 | 第 1 条规则变绿 ✓             |
| 一致性满足  | 密码 1 = 密码 2   | 第 2 条规则变绿 ✓             |
| 可提交    | 两条规则均满足      | 主按钮高亮可点                  |
| 错误     | 提交后服务端报错      | 显示红字"两次密码不一致，请重新输入"      |

### 元素构成

1. **标题**："设置登录密码"。
2. **副标题**：动态显示"为账号 +86 138 0000 0000 设置登录密码，方便下次快速登录"。
3. **密码输入行 1**：占位"请输入新密码（6-20 位）" + 👁 切换显隐。
4. **密码输入行 2**：占位"请再次输入新密码" + 👁。
5. **密码规则提示**（2 条）：
   - 圆圈图标（默认灰圈，满足时变绿✓）+ 文字"6-20 位字母、数字或符号"。
   - 同上 + "两次输入一致"。
6. **错误行**（红字）。
7. **主按钮**（通栏）："完成"。
8. **次链接**（仅注册场景）："跳过，下次再设置"。

### 操作逻辑

- 输入密码 → 实时检查规则，圆圈灰变绿。
- 点击 👁 → 切换密码可见。
- 点击主按钮 → 提交；成功 → 注册流程进 P4 / 忘记密码流程弹轻提示"密码已重置"后回 P1.2。
- 点击"跳过，下次再设置" → 直接进 P4。

### 业务规则

- 密码不能与手机号相同。
- 不强制大小写 / 数字混合。

## P-FORGOT. 忘记密码流程

不是独立页面，而是一组复用 P1.1 / P2 / P3 的步骤序列。

### 流程

1. 入口：P1.2 → 点击"忘记密码？"。
2. P1.1 子步骤（输入手机号）。
3. P2 子步骤（验证码）。
4. P3 子步骤（设置新密码）。
5. 完成：弹轻提示"密码已重置" → 自动跳回 P1.2，手机号已自动填入。

### 关键差异（与注册流程相比）

- P3 完成后不进入 P4，而是回 P1.2。
- P3 不显示"跳过，下次再设置"次链接。

## P4. 完善个人资料（子步骤）

注册后引导页，所有字段均可"稍后完善"，但昵称为必填。

### 状态

| 状态     | 触发条件               | 视觉                     |
| ------ | ------------------ | ---------------------- |
| 默认     | 注册成功首次进入           | 头像随机一个 emoji；昵称为空；三联表单为空 |
| 昵称为空   | 用户清空了昵称            | 主按钮置灰                  |
| 昵称非空   | 输入了任意昵称            | 主按钮高亮可点                |
| 加载中    | 提交后等待服务端响应         | 主按钮内 spinner             |

### 元素构成

1. **标题**："完善个人资料"。
2. **副标题**："让 AIRing 更懂你 · 资料仅你可见"。
3. **头像区**（居中）：
   - 圆形头像（默认随机 emoji，如 🦊）。
   - 下方按钮"🎲 选择头像"。
4. **昵称行**（必填）：
   - 标签"昵称*"。
   - 输入框（最多 16 字）。
   - 右侧 🎲 随机昵称按钮。
5. **三联表单**（横排 3 列）：
   - 年龄：标签 + 只读输入框 + "岁"后缀（点击触发滚轮）。
   - 身高：同上 + "cm"。
   - 体重：同上 + "kg"。
6. **健康标签区**：
   - 标签"健康标签 (可多选)"。
   - 隐私说明（带🔒）："仅用于为你提供个性化健康分析，**他人不可见**。"
   - 标签 chips 网格：高血压关注 / 糖尿病关注 / 备孕中 / 失眠 / 慢性病 / 减脂 / 增肌 / 孕期 / 哺乳期 / 老年关怀…
7. **主按钮**（通栏）："完成"。
8. **次链接**："稍后完善"。

### 操作逻辑

- 点击头像 / "选择头像" → 弹 P5.2 头像选择器。
- 输入昵称 → 主按钮联动启用 / 禁用。
- 点击 🎲 随机昵称 → 自动填入一个随机昵称。
- 点击年龄 / 身高 / 体重行 → 弹 P5.1 滚轮选择器。
- 点击标签 chip → 切换选中态（蓝填充 / 灰描边）。
- 点击主按钮 → 提交资料 → 弹 P-LOGIN-DONE 轻提示 → 1 秒后自动跳关心 Tab。
- 点击"稍后完善" → 直接跳关心 Tab，弹轻提示"可在个人设置中补充"。

### 业务规则

- 昵称同名允许重复（系统加随机后缀给唯一 ID）。
- 健康标签数量不限。
- 数据加密存储；标签不出现在关心圈。

## P5. 个人设置

### 状态

| --------- | -------------------------- | ----------------------------------- |

#### a. 顶栏

- ‹ 返回（左）+ 中标题"消息中心" + 右上文字按钮"全部已读"。

- **全部** · **好友** · **挑战** · **健康** · **系统**（每个 Tab 两字，与 v2.1 视觉收敛保持一致）。
- 每个 Tab 标题右侧带未读计数小徽章（≥ 100 显示 `99+`，= 0 隐藏徽章）。

#### c. 时间分组列表

- 三个分组小标题：**今天** / **昨天** / **更早**（11/700 灰色，间距 12px）。
- 每个分组下罗列消息卡，卡片间距 8px。当前 Tab 下若分组无内容，整个分组隐藏。

#### d. 消息卡（`.msg-item`）

每张消息卡的元素自左向右：

1. **分类彩色圆形图标**（40×40，按四类着色：好友 = 蓝色渐变 / 挑战 = 金橙渐变 / 健康 = 绿色渐变 / 系统 = 中性灰）。图标内容：UI 类用 SVG（heart / message / user-add / star / gift / info），健康类允许 emoji（💊 💧 ❤️ 😴 等，符合《00-设计规范》§ 4.2）。
2. **正文区**：

### 操作逻辑

#### a. 全局动作

- 点击 ‹ 返回 → 回上级页面（P5 或一级页面），P7 滚动位置不保留（每次进入都从顶部开始）。
- 点击分类 Tab → 切换内容区显示该分类条目，其它分类的未读徽章保持不变；切换时**不重置** P7 滚动位置（用户在"全部"中往下翻后再切到"健康"，回到"全部"仍然在原位置）。
- 点击"全部已读" → 清空所有红点 + 4 类徽章计数归零；按钮置灰；轻提示"已全部标为已读"。**不会**触发任何条目的物理删除——条目仍保留供回看，只是 unread 标记为 false。
- 进入 P7 时**不自动 mark all as read**：用户必须显式动作或单条点击才算已读，避免误把"批量收件"当成"已读"。
- P7 中任何一次 unread → read 的状态变化都会同步刷新一级页面顶栏 🔔 按钮的徽章（基于全局未读总数）。

#### b. 点击消息卡（非按钮区）的统一规则

任何消息条目，**点击非按钮区域**都会触发以下连锁反应：

1. 该条 `is-unread` 立即变 false，红点消失。
2. 当前分类 Tab 计数 -1。
3. 全局未读总数 -1，顶栏 🔔 徽章同步。
4. **push 转场跳转**到对应目的页（见下方 c. 点击映射表）。目的页通常通过 query 参数 `?focus=msg-{id}` 携带消息 ID，用于自动滚动 / 高亮目标内容。
5. 目的页顶栏左 ‹ 返回 → 回 P7，**保留 P7 离开时的滚动位置**与所选 Tab。

#### c. Z 方案分流规则（点击主体 → 三种路径之一）

唯戒按 **"消息本身的属性"** 决定点击路径，不按 4 类硬分。判定标准：

> 该消息是否存在一个"已经在 App 里、用户经常去且会想停留"的目的页？
>
> - **存在** → 走「直跳目的页」（类型 ①）
> - **不存在 / 目的页只是占位** → 走「进 P7.detail 详情页」（类型 ②），由详情页本身承载所有信息与可执行操作
> - **本质是行内决策** → 走「不跳页 · 行内动作」（类型 ③）

##### c.4 路径分流的兜底原则

- 类型 ① 与 ② 的边界判定**写在消息生成时**（后端字段 `route_type`），不允许同一类消息出现两种路径，保持一致性。

- 类型 ② 的详情页若存在"明确目的页"（如徽章可去 L1.A、戒指可去 P6.1），通过详情页底部的次按钮提供"去...看看"链接，不强制跳转。

#### d. 点击行内动作按钮的处理

行内动作按钮**就地处理、不跳页**（除"查看挑战 / 立即领取"等明确指向跳转的按钮外）。处理后条目本身 **保留** 在列表中，但视觉降级为"已处理灰态"，不再显示行内按钮，红点消失。

#### e. 几个用户体验细节

- **重复点击防护**：行内动作按钮按下后立即置灰 + 加 200ms 防抖，避免重复发起请求。
- **失败回退**：网络失败时按钮恢复可点 + toast 提示，不改变红点和计数（只有服务端确认成功后才更新）。
- **滚动位置保留**：分类 Tab 切换不重置滚动；从目的页返回 P7 时恢复离开前位置（前端用 sessionStorage 暂存 scrollTop + activeTab）。
- **键盘 / 无障碍**（待 P-TBD-14 完整化）：每条消息为独立 `<article role="listitem">`；行内动作按钮 `aria-label` 明确，标题与动作按钮可被屏幕阅读器分别朗读。

### 业务规则

- **隐私铁律**：健康类消息不暴露原始数值（HRV / 心率 / SpO2 / 步数 / 睡眠时长 / 服药剂量），仅给定性 + 建议；位置类降级到约 1 km；遵守《00-设计规范》§ 10。

- **合并规则**：同人 5 分钟内多次互动合并 1 条（与 02-关心模块 § 通知规则一致）。具体规则：
  - 5 分钟内同人多次点赞我的同一动态 → 仍然 1 条「X 给你的"Y"点赞」。
  - 5 分钟内同人多次评论 → 副文显示最新一条 + "等 N 条"。
  - 跨动态的点赞 → 不合并（每个被赞动态独立 1 条）。

- **未读三级状态机**：
  - **条目级**：每条消息有 `is_unread` 字段（boolean）。点击主体或行内动作处理后立即翻 false。
  - **分类级**：分类 Tab 计数 = 该分类下 `is_unread = true` 的条目数。计数变化即时同步。
  - **全局级**：顶栏 🔔 徽章 = 4 类未读总和（>= 100 显 "99+"，= 0 隐藏）。
  - **持久化**：已读状态服务端同步，跨设备 / 重新登录后保持。

- **已读 vs 移除（区分）**：
  - **标已读**：条目仍在列表，仅去掉红点 + 不计入未读；适用于"点击主体跳目的页"、"全部已读"、"行内动作处理"。
  - **物理移除**：条目从列表彻底删除；仅在以下场景触发：① 自动清理周期到（保留期见下）② 用户主动左滑删除（暂未规划，待 P-TBD-15）。

- **跳转上下文协议**：
  - 跳出 P7 时通过 query 参数 `?focus=msg-{id}` 或语义化参数（如 `?cid={挑战id}`、`?metric=hrv`）传递上下文。
  - 目的页解析参数后自动滚动到目标内容，并播 1.5 秒高亮动画（光晕 / 描边脉冲），结束后恢复常态。
  - 返回 P7 时还原 `(activeTab, scrollTop)`；用户离开 P7 超过 30 分钟则清除暂存，重新从顶部开始。

- **保留期**（待产品最终确认，详见 P-TBD-12）：
  - 好友 / 挑战类：30 天后自动清理。
  - 健康类：7 天后自动清理（避免堆积）。
  - 系统类：90 天；其中"抽奖结果"按 7 天领取截止后即刻进入灰态"已过期"。

- **推送开关下放**：在 P5 → 消息与通知 区段提供"通知设置"二级入口（本期作为 P-TBD-11 待规划），按 4 类分别开关「站内显示」与「系统推送」；用户关闭某类的"站内显示"后，已存在的该类条目仍可见（不溯及既往），仅停止新增。

- **未登录态**：直接重定向到 P1，登录成功后用 query 参数携带 `?from=P7` 回到 P7。

## P7.detail. 消息详情页（v2.1 新增 · 单一布局多变体）

二级页面，承载所有"类型 ② · 进详情"消息（参见 P7 § c.2）。**单一布局 + 内容区按 `detailType` 自适应**，避免每种消息单独设计页面。

### 状态

| 状态     | 触发条件                  | 关键差异                                      |
| ------ | --------------------- | ----------------------------------------- |
| 未操作    | 进入页面，未点主操作按钮          | 主按钮高亮可点；次按钮（"去...看看"等）可点                  |
| 处理中    | 点击主按钮（如领奖填地址提交后等待）    | 主按钮 spinner + 文案改"处理中..."；其它操作禁用           |
| 已完成    | 主操作处理完成（领奖成功 / 已确认等）  | 主按钮变 ✓ 文案（"已领取" / "已确认"），不可点；副文展示成功状态     |
| 已过期    | 仅 prize 变体，超 7 天领取截止 | 主按钮置灰，文案改"已过期"；附说明"奖励已退回积分账户"             |

### 元素构成

#### a. 顶栏

- ‹ 返回（左）+ 中标题（按变体动态：徽章详情 / 奖品详情 / 升级里程碑 / 戒指低电量 / 系统通知 等）+ 右"分享"图标（可选，仅徽章变体显示）。

#### b. Hero 区（统一骨架）

自上而下：

1. **大圆角彩色图标**（80×80，圆角 var(--radius-xl)）：按四类着色，与 P7 列表 `cat-*` 同源色。图标内容按变体覆写（如 prize → 礼盒图、ring-battery → 电池图、badge → 徽章 emoji）。
2. **主标题**：var(--font-2xl) / var(--weight-bold)，单行省略；文案 = 该消息的核心信息（"解锁连击 7 天徽章" / "你抽中了定制收纳盒" / "戒指电量低于 20%"）。
3. **副信息行**：var(--font-sm) / var(--color-text-secondary)；展示 **来源 + 时间**："来自挑战「30 天早睡」 · 3 小时前" / "戒指 SN AR-8F2C-1D40 · 30 分钟前"。

#### c. Body 区（按 detailType 自适应）

每种 detailType 在 Body 区渲染不同的内容卡组合，但都遵守通用容器（`.detail-section` + `.detail-section-title`）。下表列出 10 种变体的核心内容：

| detailType | 核心内容卡（自上而下） |
| --- | --- |
| `badge` | ① "解锁条件"卡（图标 + 描述）；② "已解锁徽章 12 / 48"链接卡。徽章主图 + 等级 + XP 已合并入 Hero（不重复展示） |
| `prize` | ① "收货地址"内嵌表单（姓名 / 电话 / 地址 textarea）；② 7 天倒计时卡（红字提示）。奖品主图 + 描述 已合并入 Hero（不重复展示） |
| `level-up` | ① 大数字 "Lv.12"（var(--font-4xl) 渐变）+ XP 进度条；② "本次获得 +50 XP"；③ 升级解锁内容列表（解锁挑战 / 徽章） |
| `streak` | ① 火焰大图标 + "连击 7 天" / "连击中断 第 3 天"；② 连击历史时间线；③ 提示语（保留 / 重启） |
| `ring-battery` | ① 电池大图（按当前电量填充色）+ 百分比；② 上次充电时间 + 平均续航；③ "充电指引" 3 步骤卡 |
| `ring-disconnect` | ① 戒指 + 红 X 大图；② 上次连接时间 + 信号丢失原因；③ "可能原因"列表（信号 / 电量 / 距离） |
| `firmware` | ① 当前版本 → 新版本（箭头图）；② "本次更新"列表（变化点）；③ 升级耗时预估（约 5 分钟，期间需保持靠近手机） |
| `app-version` | ① 当前版本 → 最新版本；② 更新内容列表（≤ 5 项要点）；③ "前往 App Store" CTA |
| `privacy-policy` | ① 政策更新摘要 3 ~ 5 项要点；② "完整政策（外链 H5）"次按钮；③ 生效日期 |
| `login-anomaly` | ① 登录设备 / 时间 / IP 大字；② 地图占位（约 1km 精度，遵守 § 10）；③ 安全提示（密码 / 信任设备） |

#### d. Action 区（统一在底部，sticky 或正文末尾）

通用按钮组：

1. **主按钮**（`.btn-block.btn-primary`，48 高）：按变体设定文案，详见 P7 § c.2 表格。
2. **次按钮**（`.btn-block.btn-ghost` 或行内文字按钮）：通常是"去 X 看看"跳目的页（如徽章 → L1.A、戒指 → P6.1）。
3. **第三按钮**（仅 login-anomaly 变体）：破坏性"不是我（紧急冻结）"，红色按钮。

#### e. 底部安全区

- env(safe-area-inset-bottom) 留白。

### 操作逻辑

- 点击 ‹ 返回 → 回 P7，保留 P7 离开时的 (activeTab, scrollTop)。该消息已被点开主体一次，红点已消失。
- 点击主按钮 → 按变体执行对应动作（见 P7 § c.2 主操作列）。处理中态期间禁用所有按钮 + 主按钮 spinner。
- 点击次按钮"去...看看" → push 跳目的页（同 P7 § c.1 直跳逻辑）；返回时仍回到 P7.detail（不是直接回 P7）。
- 点击表单内字段（仅 prize 变体）→ 弹键盘 / 唤起对应输入；提交后转处理中态。
- 在已完成态下再次进入（如点已领取的奖品）→ 主按钮显示 ✓ 已领取（不可点）+ 副文增加发货状态。

### 业务规则

- **进入路径单向**：P7 列表 → P7.detail；不允许从其它二级页 push 进 P7.detail（外部分享链接除外，本期不做）。
- **Body 不重复 Hero**：Hero 区已经承担"主图 + 主标题 + 关键属性 + 来源 + 时间"的展示职责，Body 区严禁再次出现同一图标 / 主标题 / 等级 + XP / 来源等信息（避免视觉冗余）。等级 / XP / 奖品材质等关键属性应合并入 Hero meta 行（多行允许）。
- **状态持久化**：`is_unread` 在 push 进 P7.detail 那一刻就翻 false（与点击 P7 列表卡同步），不要等用户在详情页 dwell N 秒。
- **prize 变体的 7 天倒计时**：精确到秒级显示，过期后刷新页面会切换到"已过期"状态；过期不会自动隐藏条目，而是改副文+按钮置灰，让用户能看到"为什么没拿到"。
- **login-anomaly 变体**：点"不是我（紧急冻结）"前必须二次确认（弹 § P5.5 同款破坏性 alert），确认后立即冻结账号 + 推送邮件 + 强制下线所有设备 → 跳 P1 重新登录。

## 业务规则汇总

### 账号体系

- 一手机号 = 一账号；不支持多手机号绑定。
- 注销账号：本期不在 P5 中提供，后续接入"账号设置 → 注销"二级页（TBD-3）。
- 设备绑定：一账号最多绑 3 个戒指，超出需先解绑。

### 数据存储与安全

- 个人资料（头像 / 昵称 / 年龄 / 身高 / 体重 / 健康标签）：服务端存储，端到端加密。
- 健康标签**仅参与服务端的健康推荐算法**，不会同步到关心圈、不会用于推荐"附近的人"。
- 密码：加盐哈希存储，永不明文回传。
- 登录凭据：1 天过期 + 30 天刷新机制。

### 协议与合规

- 用户协议、隐私政策为外链 H5 页（应用内浏览器）。
- 14 岁以下用户禁止注册（手机号注册时无法识别，主要靠出生年龄校验：> 14 岁才允许填年龄）。
- iOS 端遵循透明度规范，定位、通知、健康数据访问需独立权限弹窗。
